# O^3 — OpenShift on OpenShift (via OpenShift Virtualization)

O^3 deploys a tenant OpenShift cluster as KubeVirt virtual machines on an existing infrastructure OpenShift cluster running OpenShift Virtualization. The automation uses the **Agent-based installer** to provision the tenant cluster end-to-end.

For background on the concept and architecture, see the blog post: [Nested OpenShift Using OpenShift Virtualization](https://www.openshift.com/blog/nested-openshift-using-openshift-virtualization).

## What the Automation Does

The deploy playbook (`openshift-O^3-cluster-deploy.yaml`) performs the following:

1. Tears down any existing tenant cluster (idempotent re-deploy)
2. Loads shared and cluster-specific configuration
3. Retrieves infra cluster credentials from Bitwarden Secrets Manager
4. Downloads `oc`, `openshift-install`, and `butane` binaries
5. Generates `install-config.yaml`, `agent-config.yaml`, and OpenShift manifests
6. Creates the Agent installer ISO and stages it on a web server
7. Prepares the infra cluster (namespace, Network Attachment Definition, CA ConfigMap)
8. Deploys KubeVirt VMs for control-plane and worker nodes (booting from the Agent ISO)
9. Waits for bootstrap completion and cluster installation
10. Configures an HTPasswd OAuth identity provider

The destroy playbook (`openshift-O^3-cluster-deco.yaml`) removes the VMs, DataVolumes, PVCs, and namespace from the infra cluster.

## Prerequisites

### Infrastructure Cluster

- An OpenShift 4.x cluster with **OpenShift Virtualization** installed
- A storage class that supports dynamic provisioning (Block volume mode recommended for etcd latency). The storage class name is read automatically from a `cluster-metadata` ConfigMap on the infra cluster.
- A **network bridge** on the infra cluster worker nodes (created via NNCP — see [Infrastructure Setup](#infrastructure-setup) below)
- Cluster-admin access (kubeconfig stored in Bitwarden)

### External Services

- **DHCP** — entries for each O^3 VM MAC address with reserved IPs
- **DNS** — `api.<cluster>.<domain>`, `api-int.<cluster>.<domain>`, `*.apps.<cluster>.<domain>`, and individual node hostnames
- **Load Balancer** — ports 6443 + 22623 balanced to control-plane IPs; ports 80 + 443 balanced to worker IPs
- **Web Server** — reachable from infra cluster nodes, serves the Agent ISO over HTTP

### Control Node (where Ansible runs)

Install dependencies:

```bash
cd src/ansible
./update_env.sh
```

This installs Python packages (`requirements.txt`), Ansible Galaxy collections (`requirements.yaml`), and the Bitwarden Secrets CLI.

Key Python packages: `ansible-core>=2.20`, `kubernetes>=28.0`, `openshift>=0.13`, `hvac`, `dnspython`.

Key Ansible collections: `community.kubernetes`, `community.libvirt`, `dellemc.openmanage`.

## Infrastructure Setup

### Network Bridge (NNCP)

Create a bridge on infra cluster worker nodes so the O^3 VMs can attach to the host network. Update the interface name to match your environment:

```yaml
apiVersion: nmstate.io/v1alpha1
kind: NodeNetworkConfigurationPolicy
metadata:
  name: r730-1.arvhomelab.com-br
spec:
  nodeSelector:
    node-role.kubernetes.io/worker: ""
  desiredState:
    interfaces:
      - name: r730-1.arvhomelab.com-br
        description: Linux bridge for O^3 tenant VMs
        type: linux-bridge
        state: up
        ipv4:
          dhcp: true
          enabled: true
        bridge:
          options:
            stp:
              enabled: false
          port:
            - name: enp3s0  # change to match infra node NIC
  nodeSelector:
    kubernetes.io/hostname: r730-1.arvhomelab.com
```

Verify the bridge is configured:

```bash
oc get nncp
NAME                       STATUS      REASON
r730-1.arvhomelab.com-br   Available   SuccessfullyConfigured
```

The bridge name must match `infra_cluster_bridge_name` in your vars file. The Network Attachment Definition is created automatically by the playbook.

## Configuration

All cluster variables live under `src/ocp-cluster-vars/vars/`. Each tenant cluster has its own directory (e.g. `tenant1/`) containing three files:

### `main.yaml` — config loader

Lists the variable files to load (common, cluster-specific, and vault-encrypted credentials). No changes needed unless adding new variable files.

### `vars.yaml` — cluster-specific variables

| Variable | Description |
|----------|-------------|
| `platform` | Must be `"O^3"` |
| `base_domain` | Base DNS domain |
| `ocp_cluster_name` | Cluster name (must match the directory name) |
| `ocp_version` | OpenShift version (e.g. `"4.21"`) |
| `ocp_bin_channel` | Release channel (`stable`, `fast`, `candidate`) |
| `ocp_network_type` | CNI plugin (`OVNKubernetes`, `Cilium`, `Calico`) |
| `infra_cluster_fqdn` | FQDN of the infrastructure cluster |
| `infra_cluster_bridge_name` | Name of the bridge on infra cluster nodes |
| `infra_cluster_storage_volume_mode` | `Block` (recommended) or `Filesystem` |
| `infra_cluster_storage_access_mode` | `ReadWriteOnce` or `ReadWriteMany` |
| `agent_iso_url` | HTTP URL where the Agent ISO will be served |

### `inventory.yaml` — node definitions

Defines the `webserver`, `control`, and `worker` host groups. Each node specifies:

- **Hostname** as the inventory host (e.g. `control1.tenant1.k8s.arvhomelab.com`)
- **`nic1.mac_address`** — pre-generated MAC (must match DHCP reservation)
- **`nic1.vlan`** — VLAN tag (if applicable)

Group vars set the VM sizing:

| Variable | Control Plane | Worker |
|----------|---------------|--------|
| `cpu` | 4 | 8 |
| `memory` | 28Gi | 32Gi |
| `rhcos_os_disk_size` | 100Gi | 100Gi |
| `rhcos_etcd_disk_size` | 5Gi | — |
| `node_role` | master | worker |


### Credentials (Bitwarden)

Sensitive values (pull secret, SSH keys, infra kubeconfig, trust bundle) are retrieved at runtime from Bitwarden Secrets Manager. The vault-encrypted `credentials.yaml` stores the Bitwarden access token.

## Deploy

```bash
cd src/ansible
export CLUSTER_NAME=tenant1
ansible-playbook playbooks/openshift-O^3-cluster-deploy.yaml \
  -i ../ocp-cluster-vars/vars/$CLUSTER_NAME/inventory.yaml \
  -e cluster_name=$CLUSTER_NAME \
  --vault-password-file ./vault-password-file
```

### Useful Tags

Skip or target specific stages:

```bash
# Skip the teardown + pre-deploy stages (re-run from manifest creation onward)
--skip-tags pre-deploy

# Only run install monitoring and auth setup (after VMs are already deployed)
--tags openshift-install-time-operations,openshift-auth-provider
```

## Destroy

```bash
cd src/ansible
export CLUSTER_NAME=tenant1
ansible-playbook playbooks/openshift-O^3-cluster-deco.yaml \
  -i ../ocp-cluster-vars/vars/$CLUSTER_NAME/inventory.yaml \
  -e cluster_name=$CLUSTER_NAME \
  --vault-password-file ./vault-password-file
```

This removes all KubeVirt VMs, DataVolumes, PVCs, and the tenant namespace from the infra cluster.

## How It Works

The Agent-based installer replaces the older UPI/ignition workflow. Instead of uploading RHCOS images and injecting ignition configs via cloud-init, the automation:

1. Generates an `agent-config.yaml` with node MACs, roles, and a rendezvous IP
2. Builds a bootable Agent ISO containing the full cluster configuration
3. Stages the ISO on a web server
4. Creates KubeVirt `VirtualMachine` resources whose DataVolumes pull the ISO over HTTP as a CDROM
5. Each VM boots from the Agent ISO, discovers its peers, and self-assembles the cluster

The VM template provisions blank DataVolumes for root disks (and a separate etcd disk for control-plane nodes), using the infra cluster's storage class with the configured volume mode and access mode.
